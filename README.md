# LLM 이메일 에이전트 간접 프롬프트 인젝션 연구

> **경희대학교 컴퓨터공학과 졸업연구 (2026)**
> OpenClaw 기반 LLM 이메일 에이전트에서, 사용자가 수정 가능한 에이전트 행동 정의를 약화시켰을 때 이메일을 통한 간접 프롬프트 인젝션(IPI)이 실제 고권한 행동(메일 발송)으로 이어지는 조건을 실증 분석하고, 화이트리스트 기반 방어를 구현·검증한 연구입니다.

> 이 저장소는 코드 저장소가 아니라 **연구 아카이브**입니다. 연구의 본체는 실험 설계·과정·발견과 논문이며, 코드(`tools/`)는 실험을 위한 계측 도구입니다.

## 연구 결과물
- 최종 보고서(논문)

## 연구 개요
LLM 에이전트가 외부 콘텐츠(이메일 본문 등)를 신뢰해 처리할 때, 그 안에 숨겨진 지시가 실행되는 취약점이 간접 프롬프트 인젝션(IPI)입니다. 본 연구는 "일반 사용자가 편의를 위해 에이전트 행동 정의를 수정했을 때, 이메일 기반 간접 인젝션이 실제 메일 발송으로 이어지는가, 어떤 조건에서 발화하는가"에 답합니다.

- **연구 범위**: 사용자가 수정 가능한 행동 정의 측면 (코어 내장 방어는 사용자가 제거 불가하므로 범위에서 제외)
- **학술 기반**: Greshake et al.(2023), OWASP LLM01:2025, CrowdStrike 인젝션 기법 분류

## 실험 환경
| 항목 | 값 |
|---|---|
| 에이전트 플랫폼 | OpenClaw v0.12.0 |
| 백엔드 LLM | Gemini 계열 fallback |
| 이메일 스킬 | gog (gogcli), Google OAuth |
| 외부 채널 | Discord (채널별 세션 분리) |
| victim 계정 | 실험 전용 Gmail (본인 통제) |

**관측된 코어 방어 (사용자 제거 불가)**
1. 외부 입력 격리 — 모든 외부 메시지를 `<<<EXTERNAL_UNTRUSTED_CONTENT>>>` 태그로 감싸 전달
2. 모델 내재 안전 행동 — 외부 메일의 발송 지시에 확인 요구 또는 거부

## 실험 설계
- **보안 조건(독립변수)**: `baseline` → `round_md1~3`(행동 정의 단계적 약화) → `round_guard`(방어 적용), 한 번에 하나만 변경(ablation)
- **인젝션 기법 3군**: Overt(명시형) / Social(사회·인지형) / Evasive(회피형) — 목표 행동은 고정, 기법만 변형
- **명령 프레이밍**: "요약해줘" vs "처리해줘"
- **반복 측정 단위**: Discord 채널 (LLM 비결정성 통제, 라운드마다 새 채널)
- **trace 단계(S0–S7)**: 수신 → gog read → 프롬프트 구성 → 모델 응답 → 발송 의도 → tool routing → 승인 게이트 → 발송 실행

## 핵심 발견
- **발화 조건**: 인젝션은 단순히 보안 문구가 제거될 때가 아니라, 에이전트가 외부 입력을 신뢰하고 즉시 처리하도록 지시받았을 때 발화한다.
- **확률적 성격**: 동일 조건에서도 발화 여부가 갈리며, 방어는 결정론적이지 않고 확률적·조건 의존적으로 작동한다.
- **세션 상태 가설**: 한 채널에서 한 번 발화하면 같은 세션 내에서 계속 발화하는 경향을 관찰.

## 방어 구현 (Guard mode)
발송 단계에서 `gogcli` 호출을 가로채는 화이트리스트 기반 shim을 삽입해, 인가되지 않은 외부 주소로의 발송을 차단하고 재확인을 요구. `round_guard`에서 인젝션이 발화하더라도 데이터 유출을 차단함을 검증.

### 도구 참고 — 분석 방법의 전환
`trace_analyze.py`는 trace 로그 자동 집계를 시도했으나, 키워드 기반 판정의 한계로 S4/S5 단계가 부정확하게 오판되었습니다. 최종 결론은 이 자동 분석기가 아니라 세션 transcript의 실제 모델 응답과 shim의 발송 로그를 직접 검증해 도출했습니다. 자동화의 한계를 확인하고 더 신뢰할 수 있는 수동 검증으로 전환한 판단 과정을 그대로 남겨 둡니다.
